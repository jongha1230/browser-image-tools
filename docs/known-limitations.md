# Browser Support And Known Limitations

## 공개 검증 범위

- 2026년 3월 16일 기준 공개 수동 확인은 Chromium 계열 브라우저에서 현재 라우트 로딩과 기본 처리 흐름을 중심으로 진행했습니다.
- 최신 Chrome, Edge 같은 Chromium 계열 데스크톱 브라우저를 우선 권장합니다.
- Safari와 Firefox도 필요한 웹 API가 모두 동작하면 사용할 수 있지만, 현재 저장소에서 공개적으로 수동 검증 범위를 넓게 약속하지는 않습니다.

## 이 제품이 전제하는 브라우저 기능

- Canvas
- Blob
- `URL.createObjectURL`
- 다운로드 링크(`a[download]`)
- Web Worker

위 기능 가운데 일부가 빠지거나 제한되면, 결과 저장이나 백그라운드 처리 같은 단계가 느려지거나 실패할 수 있습니다.

## 워커 폴백 동작

- 백그라운드 작업자(Web Worker)를 우선 시도합니다.
- Worker 초기화가 안 되거나 `OffscreenCanvas`, `convertToBlob` 같은 관련 API가 빠진 환경에서는 메인 스레드로 같은 처리 흐름을 이어갑니다.
- 이 경우 결과 형식과 저장 흐름은 유지되지만, 처리 중 탭 스크롤과 입력 반응이 더 느려질 수 있습니다.

## 현재 제한 사항

- 입력과 출력은 JPEG, PNG, WebP만 지원합니다.
- PDF, HEIC, RAW, 비디오, 서버 업로드, 계정, DB, 클라우드 저장은 현재 범위에 포함하지 않습니다.
- 모든 처리는 브라우저 메모리 안에서 이뤄지므로 매우 큰 파일, 긴 배치, 반복 ZIP 내보내기에서는 느려지거나 부분 실패가 생길 수 있습니다.
- 모바일 브라우저는 데스크톱보다 메모리 한계에 더 빨리 닿을 수 있어 큰 작업에서는 덜 안정적일 수 있습니다.
- 브라우저가 특정 형식 재인코딩을 지원하지 않으면 일부 출력 형식은 실패할 수 있으며, 이 경우 UI에서 파일별 오류 메시지를 표시합니다.
- 탭을 새로고침하거나 닫으면 현재 큐, 결과, 다운로드 준비 상태는 유지되지 않습니다.
- EXIF 제거는 재저장 방식이라 메타데이터 정리에 도움이 되지만, 이미지 안에 직접 보이는 민감 정보까지 숨기지는 않습니다.
- JPEG와 WebP는 재인코딩 품질 설정의 영향을 받고, PNG는 무손실 재저장 특성상 기대보다 파일이 더 커질 수도 있습니다.

## 실무적으로 권장하는 사용 방식

- 큰 배치 전에 대표 파일 1장으로 먼저 결과를 확인합니다.
- 사진, 캡처, 투명 배경 로고를 한 큐에 섞기보다 용도별로 나눕니다.
- 큰 작업은 모바일보다 데스크톱 브라우저에서 먼저 시도합니다.
- ZIP 다운로드를 끝내기 전에는 탭을 닫거나 새로고침하지 않습니다.

## 범위 밖 항목

- 백엔드 업로드
- 계정, 인증, 데이터베이스
- 클라우드 저장 또는 동기화
- PDF, HEIC, RAW, 비디오 처리
- 엔드유저 대상 AI 기능

## 후속 개선 메모

- 브라우저별 수동 QA 범위를 Chromium 밖으로 확장
- 대용량 배치 처리 시 메모리 사용량과 진행률 체감 개선
- 실제 광고 운영 정보가 준비된 뒤 정책 고지와 관련 기술 파일 보강

증빙 캡처와 아키텍처 요약은 [docs/proof-assets.md](./proof-assets.md)에서 함께 확인할 수 있습니다.
